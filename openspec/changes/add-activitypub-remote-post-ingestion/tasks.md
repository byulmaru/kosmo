## 1. Data Model

- [ ] 1.1 ActivityPub object URI와 kosmo `Post`를 연결하는 `activitypub_object` 테이블을 추가하고 `id`, `uri`, `type`, `activityPubActorId`, `postId`, `receivedAt`, nullable `publishedAt`을 저장한다.
- [ ] 1.2 `ActivityPubObjectType.NOTE`, `TableDiscriminator.ActivityPubObjects`, `uri` unique, `postId` unique, `activityPubActorId -> activitypub_actor.id` non-unique indexed foreign key, `postId -> post.id` foreign key를 추가하고 actor URI를 중복 저장하거나 object type에 unique constraint를 두지 않는다.
- [ ] 1.3 remote post materialization에 필요한 Drizzle table/relations/migration fixture를 갱신한다.

## 2. Remote Post Materialization

- [ ] 2.1 actor-scoped inbox와 shared inbox가 verified typed `Create`를 unsupported endpoint 404가 아니라 Fedify inbox listener를 거쳐 post materialization handler로 연결하고, 같은 activity ID 재전달은 Fedify inbox idempotency에 맡긴다.
- [ ] 2.2 verified typed `Create.actorIds`의 서로 다른 URL `.href`를 정확히 하나로 제한하고, 해당 `activitypub_actor`가 가리키는 remote `Profile.state = ACTIVE`이며 instance kind가 `ACTIVITYPUB`인지 object hydration 전에 검증한다. 서로 다른 `objectIds`의 `.href`도 정확히 하나로 제한한 뒤 `Create.getObject({ documentLoader: ctx.documentLoader })`로 embedded/IRI-only를 같은 경계에서 resolve한다. Fedify의 default `crossOrigin: "ignore"`를 유지하고 `"trust"`를 사용하지 않으며, 복수 object, hydration 실패, non-Note object는 skip하고 custom HTTP fetch/parser를 구현하지 않는다.
- [ ] 2.3 resolved Note `attributionIds`의 서로 다른 URI가 정확히 하나이고 `Create.actorIds`의 단일 URI와 같은지 검증한다. `toIds`/`ccIds`/`replyTargetIds`로 추가 hydration 없이 public top-level Note를 판정하고, 서로 다른 activity ID의 duplicate ActivityPub object URI upsert 정책을 구현한다. 최초 `Post`/`PostContent`/`activitypub_object` 생성은 하나의 transaction으로 수행해 unique conflict가 부분 row를 남기지 않게 한다.
- [ ] 2.4 content가 있으면 Note `mediaType`을 parameter와 ASCII case에 무관한 MIME essence로 정규화하고 absent는 `text/html`로 취급한다. `@tiptap/html`을 `@kosmo/core` dependency로 추가하고 `@kosmo/core/tiptap`에 server-side HTML helper를 만들어 `text/html` content를 기존 `Document`/`Paragraph`/`Text` extensions로 `generateJSON()`한 뒤 canonical `bodyJson`으로 정규화하고 여기서 `bodyText`를 추출한다. `text/plain`도 기존 plain-text helper로 canonical `bodyJson`을 먼저 만든 뒤 `bodyText`를 추출한다. malformed/unsupported MIME essence나 TipTap 변환 실패는 부분 row 없이 skip하며, content가 없거나 attachment-only인 Note는 attachment를 저장하지 않고 빈 본문으로 materialize하고 모든 remote HTML 원본은 `bodyHtml`에 저장하지 않는다.
- [ ] 2.5 remote Note를 `ACTIVE` `Post`와 `PostContent`로 materialize한다. Note `published` 또는 최초 수신 시각 fallback을 `Post.createdAt`으로 저장하되 5분을 초과한 미래 published는 수신 시각을 사용하고 object mapping에는 원본을 보존하며, 각 최초/변경 `PostContent.createdAt`은 delivery 수신 시각으로 저장한다.
- [ ] 2.6 중복 object URI 재전달 시 기존 object mapping의 `activityPubActorId`와 이번 delivery actor URI로 조회한 `activitypub_actor.id`가 같을 때만 기존 `Post`를 재사용한다. canonical `bodyJson`의 구조가 변경되면 새 `PostContent` revision과 `currentContentId` 교체로 반영하고 같으면 기존 revision을 재사용하며, 파생 `bodyText`는 revision identity로 사용하지 않는다. 최초 visibility, object mapping의 수신 시각과 원본 published 시각 및 `Post.createdAt`은 갱신하지 않는다.
- [ ] 2.7 저장되지 않았거나 inactive/non-ActivityPub profile의 actor delivery는 object hydration 전에 skip하고 WebFinger lookup 또는 actor materialization을 수행하지 않는다. 저장된 stale actor는 DB row를 그대로 사용해 actor profile refresh를 예약/수행하지 않되, inbox signature/key verification과 accepted 단일 `Create.object` hydration은 Fedify protocol boundary에 맡긴다.
- [ ] 2.8 shared inbox로 들어온 `Create`에서 Fedify가 resolve한 단일 public Note는 local follow 관계나 recipient 검증 없이 actor attribution과 public top-level Note 검증만으로 materialize한다.
- [ ] 2.9 `SUSPENDED` instance의 delivery는 object hydration 전 차단한다. `UNRESPONSIVE` instance는 Fedify의 accepted `Create.object` hydration을 허용하고 known actor의 verified object가 `Note`로 resolve되면 materialization eligibility와 별개로 instance를 `ACTIVE`로 갱신하되, post side effect는 나머지 검증을 통과한 Note에만 수행한다.

## 3. GraphQL Post API

- [ ] 3.1 기존 origin 공통 `Profile.posts` resolver가 remote 전용 분기나 remote fetch 없이 materialized posts를 기존 ordering/pagination primitive로 반환하는지 확인하고, 필요한 instance visibility 조건만 공통 경계에 추가한다.
- [ ] 3.2 origin 공통 `Post` visibility access가 remote materialized posts를 local posts와 같은 visibility 규칙으로 다루되 작성자 instance가 `SUSPENDED`인 remote post는 노출하지 않도록 정렬한다.
- [ ] 3.3 `PostContent` Node 직접 조회가 parent `Post.state = ACTIVE`인 경우 current와 historical revision을 모두 지원하고, historical revision에는 `Post.currentContentId` 일치를 요구하지 않되 parent visibility와 작성자 profile/instance visibility를 적용하도록 정렬한다.
- [ ] 3.4 기존 origin 공통 `homeTimeline` resolver가 established `ProfileFollow`의 remote followee materialized posts를 별도 remote 분기 없이 포함하는지 확인하고, `SUSPENDED` instance 제외 조건만 공통 경계에 추가한다.
- [ ] 3.5 GraphQL schema shape가 바뀌지 않는지 확인하고 remote `Profile.posts`, PostContent history와 home timeline contract를 검증한다.
