# PR Writing Memory

## Purpose

- PR 제목, 본문, Draft/Ready 상태, stack 의존성 설명을 작성하거나 수정할 때 적용한다.
- 기본 운영 규칙은 먼저 `memory/commit-pr.md`를 따른다.

## PR Scope

- PR 하나에는 리뷰어가 독립적으로 이해하고 검증할 수 있는 하나의 기능적 변화만 담는다.
- 큰 작업은 선행 정리 PR, 기능 PR, 후속 PR로 나눌 수 있는지 검토한다.
- 선행 정리 PR에는 이름 변경, 타입 정리, 구조 이동, 기계적 리팩터링을 담는다.
- 기능 PR에는 실제 동작 변화를 담는다.
- 후속 PR에는 관측성, 문서화, 추가 최적화, 확장 대응을 담는다.
- "어차피 같은 화면/같은 기능"이라는 이유만으로 변경을 합치지 않는다. 배포와 롤백 관점에서 최소 정상 동작 단위인지 확인한다.
- 리뷰어가 로컬에서 쉽게 검증할 수 없다면 분리가 덜 된 것이다.
- stacked PR에서 특정 concern의 변경이 다른 PR 책임이면 해당 PR로 옮긴다.
- 다른 PR 위에 쌓이거나 먼저 merge되어야 하는 의존성이 있으면 PR 본문, 댓글, 또는 리뷰 응답에서 PR 번호를 명시한다.

## PR Title

- PR 제목은 한국어로 작성한다.
- PR 제목은 작업 시작 시 확인한 사용자 의도를 가장 먼저 반영한다.
- PR 제목에는 `[codex]` 같은 agent/tool 출처 prefix를 넣지 않는다.
- PR 제목은 구현 수단이나 변경 파일 목록이 아니라 PR의 의도를 기준으로 적는다.
- 제목만 봐도 "왜 이 변경이 필요한지"와 "무엇을 가능하게 하려는지"가 보여야 한다.
- 가능하면 사용자/시스템 관점에서 달성하려는 결과를 쓴다.
- 내부 구현 나열, 추상적인 정리 표현, 맥락 없는 `refactor`, `fix stuff`는 피한다.
- 좋은 예시는 `요청 간 렌더링 상태가 섞이지 않게 한다`, `배포 환경에서 서버 엔트리가 누락되지 않게 한다`이다.

## PR Body

- PR 본문은 한국어로 작성한다.
- PR 본문은 `무엇을 변경했는지`, `왜 변경했는지`, `이번 PR의 주요 결정`, `어떻게 확인할 수 있는지`, `아직 어떤 문제가 남았는지` 순서를 기본으로 한다.
- `무엇을 변경했는지`에는 리뷰어가 diff를 열기 전에 큰 구조를 파악할 수 있도록 구체적인 변경 내용을 bullet로 적는다.
- `왜 변경했는지`에는 작업 시작 시 확인한 사용자 의도, 변경이 필요한 배경, 해결하려는 문제, 줄어드는 사용자 문제나 운영 문제나 개발 흐름 문제를 설명한다.
- `이번 PR의 주요 결정`에는 구현 중 사람이 내린 중요한 선택을 결정별로 기록한다. 관찰 가능한 행동, 데이터,
  보안, 호환성, rollout, 가역성 또는 후속 구현 방향에 영향을 주며 실질적인 대안이 있었던 선택을 중요한
  결정으로 본다.
- 각 주요 결정에는 결정자가 선택한 방식, 고려한 대안, 선택 이유와 감수한 결과를 포함한다. AI는 계획·구현
  중 사람이 확정한 선택과 이유를 바탕으로 PR 결정문을 작성할 수 있다. 근거가 없거나 불명확한 선택과 이유는
  추론해 만들지 않고 결정한 사람에게 확인한다.
- 최신 canonical 문서와 Linear를 독립 대조해 유효함을 확인한 OpenSpec durable decision은 전문을 반복하지
  않고 해당 decision과 이 PR의 적용 결과를 링크한다. 구현 중 새로운 durable decision이 필요해졌다면 PR에만
  남기지 않고 적용되는 canonical 문서와 Linear upstream source를 먼저 갱신한 뒤 OpenSpec을 갱신한다.
- 새로 내린 중요한 결정이 없다면 `없음. 최신 canonical·Linear와 대조한 OpenSpec 결정을 변경 없이 적용했다.`처럼
  명시한다. 사소한 구현 선택이나 대안이 없었던 기계적 변경을 형식적인 결정으로 만들지 않는다.
- `어떻게 확인할 수 있는지`에는 자동 테스트, 수동 검증, 로컬 실행 명령을 적고, 확인하지 못한 항목은 이유와 함께 명시한다.
- `아직 어떤 문제가 남았는지`에는 설계상 열려 있는 질문, 후속 작업, 회귀 위험, 운영 영향, 의도적으로 남긴 제한을 적는다.
- 남은 문제가 없으면 `없음`이라고 적는다.

주요 결정은 다음 형식을 기본으로 한다.

```md
### <결정 제목>

- 결정자: @작성자
- 선택: <최종 선택>
- 고려한 대안: <비교한 실질적인 대안>
- 이유: <이 선택이 현재 제약과 목표에 더 적합한 이유>
- 감수한 결과: <포기한 장점, 추가된 제약 또는 후속 비용>
- 관련 근거: <canonical 문서, 최신 Linear, OpenSpec decision과 적용 결과 링크>
```

## Draft PR Policy

- 작업 초반이라도 방향성이 잡히면 Draft PR을 가볍게 연다.
- Draft PR의 목적은 승인 요청이 아니라 맥락 공유, 조기 피드백, CI 가시성 확보다.
- Draft PR에는 지금까지 된 것, 아직 안 된 것, 막히는 점 또는 확인이 필요한 점을 간단히 적는다.
- Ready for review로 전환할 때는 최소한 브랜치 HEAD가 정상 동작하고, PR 범위가 리뷰 가능한 단위인지 다시 확인한다.
